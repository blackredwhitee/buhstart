Связи между разделами

Главная → Услуги

Блок услуг на главной — карточки с кнопкой «Подробнее» → /uslugi/[slug]. Якорь #uslugi доступен из хедера.

Калькулятор → Форма

После 9 шагов калькулятора результат показывает рекомендованные услуги + кнопку «Обсудить с бухгалтером» → inline-форма с данными из расчёта.

Статьи → Услуги

В конце каждой статьи — блок «Нужна помощь?» с CTA на релевантную услугу или форму.

Услуга → Калькулятор

На страницах услуг — кнопка «Рассчитать стоимость» → /calculator с предвыбранным параметром услуги через URL-параметр.

SEO-замечания

  • ЧПУ — только кирилл. транслитерация, не id
  • sitemap.xml генерируется автоматически Next.js + Sanity
  • Все страницы услуг — SSG (статическая генерация), блог — ISR (revalidate: 3600)
  • Мета-теги и OG-изображения управляются через Sanity Studio

/calculator — Калькулятор

Блок 1 Мини-hero

H1 «Узнайте, какие услуги вам нужны». Подзаголовок: «9 вопросов — и мы подберём оптимальный пакет под вашу ситуацию». Без упоминания цен.

Блок 2 Wizard (9 шагов)

Прогресс-бар вверху. Один вопрос на экран. Анимация slide-in при переходе между шагами.

Шаг 1: Форма собственности (ИП / ООО) — крупные карточки-кнопки
Шаг 2: Система налогообложения (УСН 6% / УСН 15% / ОСНО / АУСН / Патент)
Шаг 3: Вид деятельности (из выпадающего списка + «Другое»)
Шаг 4: Количество операций в месяц (слайдер: <30 / 30–100 / 100–300 / >300)
Шаг 5: Число сотрудников (0 / 1–5 / 6–15 / 16+)
Шаг 6: Нужен ли кадровый учёт? (Да / Нет / Не знаю)
Шаг 7: Нужен ли управленческий учёт? (Да / Нет / Не знаю)
Шаг 8: Есть ли ВЭД? (Да / Нет)
Шаг 9: Стадия бизнеса (стартую / работаю до 1 года / работаю 1+ лет)

Блок 3 Экран результата

Список рекомендованных услуг (без цен). Нейтральный заголовок: «Для вашей ситуации подойдут:».

Под списком — форма: Имя + Телефон + кнопка «Получить точный расчёт». Лид → Битрикс24 с полными данными расчёта.

03

Главная страница — подробно

Каждый блок от первого экрана до футера
Контекст: что должна сделать главная страница

Посетитель пришёл не знать, чем занимается компания — он уже примерно знает. Он пришёл понять: «Это мой вариант? Им можно доверять? Что делать дальше?» Каждый блок отвечает на один из этих вопросов.

01 HERO — первый экран (100vh)

Контент

Eyebrow: «Бухгалтерское сопровождение в Москве»

H1: «Ваш бизнес — наша зона ответственности» (или «Бухгалтерия без ошибок и штрафов»)

Подзаголовок (1-2 строки): «Профессиональная бухгалтерия для ИП и ООО. Работаем удалённо по всей России. Отвечаем в течение 2 часов.»

Кнопки: «Рассчитать стоимость» (primary, оранжевый) + «Записаться на консультацию» (secondary, обводка)

Доверительные метки: ряд из 3-4 иконок со значениями — «12 лет на рынке» / «350+ клиентов» / «Ответим за 2 часа» / «Гарантируем возврат переплат»

Визуал

Правая половина: живое фото главного бухгалтера или всей команды (не стоковое). Лёгкий фон: тёплый белый с еле заметной текстурой. Navy-блок слева (20%) для контраста, если используется split-layout.

CTA: рассчитать / консультацияЭмоция: «Я в правильном месте»
02 ДОВЕРИТЕЛЬНАЯ ПОЛОСА (trust bar)

Узкий горизонтальный блок (80px высота). Фон: чуть темнее страницы. Логотипы клиентов или партнёров (если есть), или 4-5 иконок с мини-текстом: «1С:Партнёр» / «Член ПНО» / «Страхование ответственности» / «НДС, ВЭД, ОСНО».

Если логотипов клиентов нет — строка с key numbers без иконок: «Работаем с 2012 · 350+ клиентов · 0 штрафов по нашей вине»

03 УСЛУГИ — обзорный блок (anchor #uslugi)

Заголовок секции: «Что мы делаем»

Сетка: 3 колонки × 2 строки = 6 карточек услуг.

Карточка: тематическая иконка (монолайн, navy), название, 2-3 строки описания, ссылка «Подробнее». Hover: оранжевая подчёркивающая линия снизу карточки + лёгкий подъём.

Ниже сетки: текстовая ссылка «Все услуги →» на /uslugi.

CTA: подробнее по каждой + все услугиЭмоция: «Они закрывают все мои задачи»

07 ОТЗЫВЫ — интеграция Яндекс.Карты

Заголовок: «Что говорят клиенты»

Горизонтальный слайдер с карточками отзывов. Каждая карточка: аватар (первая буква имени на цветном фоне), имя, рейтинг (звёзды), дата, текст отзыва.

Справа от слайдера (или снизу) — виджет со средним рейтингом: большое число (4.9), N отзывов, логотип Яндекс.Карт, кнопка «Все отзывы».

Техническое решение: отзывы подгружаются через сервер (Next.js Route Handler) от Яндекс API или парсинг — клиент получает готовые данные.

Цветовая палитра

Палитра выводится напрямую из логотипа компании: тёплый оранжевый + нейтральный серый + белый. Никакого navy как основного цвета — он уходит в прошлое. Серый — главный нейтральный, оранжевый — единственный акцент.

Визуальная концепция

Ключевое слово

«Надёжный партнёр»

Не «строгая бухгалтерия», не «технологичный стартап». Компания, которой доверяют деньги, налоги и спокойствие.

Эмоциональная цель

Посетитель должен почувствовать: «Здесь профессионалы, которые разбираются. Мне не нужно самому вникать — они всё сделают правильно.»

Минимализм vs. насыщенность

Средний минимализм — не аскетичная пустота, но и не перегруженная страница с эффектами. Достаточно воздуха, чтобы контент дышал. Достаточно деталей, чтобы страница выглядела сделанной с вниманием.

Правило: на один экран — один визуальный акцент. Не три кнопки, не пять цветов, не анимации везде.

Фото vs. иллюстрации

  • Фото реальной команды: обязательно на главной (hero), странице /team, карточках статей (авторы). Создаёт доверие.
  • Иллюстрации: для обложек статей, декоративных фонов секций, иконок услуг. Flat-стиль, строго в палитре бренда.
  • Стоковые фото людей: запрещены. Исключение — если нет фото команды, используем нейтральные офисные детали (стол, блокнот, калькулятор) без людей.

Калькулятор на мобильном

  • Каждый шаг — полноэкранный (100vh). Вопрос вверху, варианты ответа занимают 60% экрана.
  • Кнопки ответа: высота минимум 52px (thumb-friendly).
  • Прогресс-бар: тонкая линия сверху экрана (4px высота).
  • «Назад» — иконка ← в левом верхнем углу.
  • Слайдер (шаг 4, количество операций): заменяется на 4 кнопки-варианта (слайдеры неудобны на сенсорных экранах).
  • Экран результата: список услуг (крупно), форма под ним, всё на одном экране с прокруткой.

Страница статьи на мобильном

  • Sidebar с оглавлением — скрыт. Заменяется на кнопку «Оглавление» (иконка ≡) в sticky mini-header статьи, открывает sheet снизу.
  • Sticky CTA «Записаться» из sidebar — становится частью фиксированной нижней панели.
  • Контент: font-size 16px, line-height 1.7, max-width 100%. Хорошая читаемость без горизонтального скролла.

Touch-специфика

  • Минимальный tap target: 44×44px (рекомендация Apple/Google). Кнопки, ссылки, radio — все увеличены до этого минимума.
  • Горизонтальный скролл (команда, услуги, отзывы): scroll-snap-type: x mandatory. Индикатор: точки снизу или «2/4» счётчик.
  • Hover-состояния не используются для мобильного (нет курсора). Визуальная обратная связь только через tap (active state, 100ms).
  • Яндекс.Карты: на мобильном вместо iframe — ссылка «Открыть в Яндекс.Картах» + статичный скриншот карты.

Производительность

  • LCP цель: <2.5s на мобильном 4G. Hero-изображение: preload, WebP, width 800px.
  • Шрифты: Inter через next/font (нет layout shift). Playfair Display — только если используется в H1 hero, иначе Inter везде.
  • Lazy-load все изображения ниже hero. Intersection Observer для анимаций появления.
  • Калькулятор: JS-бандл разбит на чанки. Шаги загружаются по мере прохождения (code splitting).

Существующий калькулятор — адаптация для клиента

Не создаём новый — переиспользуем готовую бизнес-логику
Ключевое требование: клиентский калькулятор на сайте — это UI-обёртка над тем же движком, что уже работает у менеджеров. Логика расчёта одна. Различается только то, что показывается на экране.

Что такое существующий калькулятор

Инструмент менеджера (файлы index.html + script.js) — 12-шаговый квиз, который:

  • Собирает параметры клиента (форма, налоги, деятельность, сотрудники, доп.услуги)
  • Рассчитывает стоимость трёх пакетов: Базовая / Стандарт / Оптима
  • Генерирует КП, Оферту и Счёт в формате .docx

Функция calcTotal() в script.js — это единое ядро расчёта. Именно её логика должна работать и в клиентском калькуляторе на сайте.

Три тарифных пакета

Базовая

Ниша + форма собственности (ООО +10к) + налогообложение + НДС + сотрудники + касса + ВЭД + доп.услуги

Клиенту: список включённых услуг. Цена: скрыта.

Стандарт

Базовая + приоритетный менеджер (+20%) + налоговый менеджмент (+20к) + бухгалтер в офисе

Клиенту: «Стандарт включает…» без цен. Менеджер добавляет надбавки при звонке.

Оптима

Стандарт + управленческий учёт (+60к)

Клиенту: показывается если ответил «да» на вопрос об управленческом учёте.

Передача данных в Битрикс24 и менеджеру

После отправки формы клиентом — Next.js Route Handler выполняет два действия:

1. Лид в Битрикс24 (crm.lead.add)

Передаётся полный объект ответов клиента в поле COMMENTS как JSON:

{
  entity: "ООО",
  tax: "osno",
  niches: ["services"],
  staffRf: "rf_1_3",
  ved: true,
  ...
  clientName: "Иван",
  phone: "+7 999 123-45-67"
}
2. URL для открытия в калькуляторе менеджера

В карточке лида — ссылка «Открыть в калькуляторе». Параметры из Битрикс24 передаются как URL-params, калькулятор менеджера их читает при загрузке и предзаполняет все шаги:

index.html?entity=ООО&tax=osno
  &niches=services&staffRf=rf_1_3
  &ved=true

Менеджер видит уже заполненный квиз — добавляет скидку, приоритет, офисного бухгалтера и формирует КП.

Что нужно реализовать в script.js для поддержки предзаполнения

Добавить функцию чтения URL-параметров при старте — она уже частично предусмотрена архитектурой (объект A с дефолтными значениями). Достаточно добавить:

// В начало index.html — после загрузки DOM
function loadFromURL() {
  const p = new URLSearchParams(location.search);
  if (p.get('entity'))   A.entity  = p.get('entity');
  if (p.get('tax'))      A.tax     = p.get('tax');
  if (p.get('niches'))   A.niches  = p.get('niches').split(',');
  if (p.get('ved'))      A.ved     = p.get('ved') === 'true';
  // ... и так для каждого поля объекта A
  if (p.size > 0) startQuiz(); // сразу открыть квиз с предзаполненными данными
}

Что дизайнер должен учесть

  • UX квиза на сайте повторяет логику шагов существующего калькулятора — тот же порядок вопросов, та же условная ветка для маркетплейсов и налогового менеджмента
  • Визуальный язык отличается: вместо компактных карточек менеджерского инструмента — крупные fullscreen-шаги (см. раздел 02 /calculator)
  • Вопрос про нулевую отчётность (Step 2) — оставить, это важная ветка: для нулёвки весь остальной квиз пропускается
  • Вопросы про воинский учёт / лицензии / СПОТ (Step 9) — на клиентском сайте можно объединить в один шаг «Дополнительные задачи» с чекбоксами
  • Скидка (Step 12) — только у менеджера. На сайте её нет и быть не должно
